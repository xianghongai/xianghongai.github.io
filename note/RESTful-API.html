RESTful

Representational State Transfer， 统一资源标识/定位(URI/L:Uniform Resource Identifier/Locator)中实现资源的呈现和状态转化；

RESTful API

示例：客户 ⇄ 订单 ⇄ 商品项；

注意资源之间的上下文关系；如 商品项 组成 订单，客户 拥有 订单，上下文体现出资源的层次结构(位置)；

# 在集合资源(Collection)节点上 `GET`(检索所有)|`POST`(创建)|`PUT`(批量更新)|`DELETE`(批量删除)；
# 在单个资源(Single)节点上，不能 `POST`(Error)；
https://www.shop.com/customers GET|POST|PUT|DELETE
https://www.shop.com/customers/:id GET|PUT|DELETE
https://www.shop.com/customers/:id/orders GET|POST
https://www.shop.com/customers/:id/orders/:id GET|PUT|DELETE
https://www.shop.com/customers/:id/orders/:id/lineitems GET|POST
https://www.shop.com/customers/:id/orders/:id/lineitems/:id GET|PUT|DELETE


https://www.shop.com/orders/:id/lineitems/:id # 通过‘订单编号’检索‘订单’，无需‘客户编号’

集合资源，使用复数名词；

单一资源，使用单数名词；如每个资源只有一个配置 configuration，它没有 ID，则不能用 POST，只能是 GET|PUT|DELETE；

资源通过 url 自描述，没有任何硬性规定，只需按逻辑确保层次结构，实现资源 URI 标识具有可理解性、可用性、可预测性；

Provider 通常无法预知数据的边界，但可以理清数据的结构关系，为 Consumer 提供有意义的内容；

基于数据消费方设计，而非数据；

RESTful API Http Request/Response

HTTP headers

Request/请求 ⇄ Response/响应；

Client Request 和 Server Response 通过 HTTP 交换信息时，报文可携带 Header 报头传递附加信息；

  • General header， 可用于 Request 和 Response，如 Date, Cache-Control, Connection；
  • Request header， 和主体(Content)无关的，用于描述 Request 信息，
    如允许条件执行 Request：Accept, Accept-*, If-*,
    如描述了 Request 本身以确保服务端能返回正确的 Response：Cookie, User-Agent, Referer
  • Response header，和主体(Content)无关的、用于描述 Response 信息；
  • Entity header，可用于 Request 和 Response，如 Content-Length, Content-Language, Content-Encoding；

HTTP Request Methods

使用 HTTP 方法，易于理解：

CRUDHttp RequestDescription
READGET从服务器获取资源
READHEAD从服务器获取资源元数据(哈希值、更新时间)
CREATEPOST在服务器创建一个资源(视需求返回新生成的资源对象)
UPDATEPUT更新服务器资源(完整)
UPDATEPATCH更新服务器资源(部分属性)
DELETEDELETE从服务器删除资源
OPTIONSOPTIONS客户端根据自己权限试探对资源能做什么操作
  • GET：无副作用，幂等
  • PUT：副作用，幂等
  • PATCH：副作用，非幂等
  • POST：副作用，非幂等
  • DELETE：副作用，幂等

如果客户端受限，HTTP Request POST 时在 Header 中添加 X-HTTP-Method-Override 指定请求的方法类别(PUT/PATCH/DELETE)；

HTTP Status Response Codes

客户端的每一次 Request，服务器都必须给出 Response；

Response 包括 Header(状态码)和 Content(数据)两部分；

HTTP Status Response Codes 有5大类100多种，

  • 1xx：相关信息
  • 2xx：操作成功
  • 3xx：重定向
  • 4xx：客户端错误
  • 5xx：服务器错误

客户端只需查看状态码，就可以判断出会话发生了什么情况，服务器需要返回精确的状态码；

2xx 状态码，操作成功

CodesDescription
200 OKGET，表示完成了资源获取
201 CreatedPOST，表示生成了新的资源
200 OKPUT，表示完成了资源更新
200 OKPATCH，表示完成了资源更新
202 Accepted异步操作，表示服务器已经收到请求，但还未进行处理，会在未来再处理
204 No ContentDELETE，表示资源已经不存在

3xx 状态码，重定向

CodesDescription
303 See Other异步操作，表示暂时重定向，用于 POST、PUT、DELETE 请求，浏览器不自动跳转，用户自己决定下一步操作，Header 中需要有 Location 字段

4xx 状态码，客户端错误

CodesDescription
400 Bad Request服务器不理解客户端的请求，未做任何处理
401 Unauthorized用户未提供身份验证凭据，或者没有通过身份验证
403 Forbidden用户通过了身份验证，但是不具有访问资源所需的权限
404 Not Found所请求的资源不存在，或不可用
405 Method Not Allowed用户已经通过身份验证，但是所用的 HTTP 方法不在他的权限之内
409 Conflict请求操作有冲突
410 Gone所请求的资源已从这个地址转移，不再可用
412 Precondition Failed客户端请求前提条件错误
415 Unsupported Media Type客户端要求的返回格式不支持。如，API 只能返回 JSON 格式，但是客户端要求返回 XML 格式
422 Unprocessable Entity客户端上传的附件无法处理，导致请求失败
429 Too Many Requests客户端的请求次数超过限额

5xx 状态码，服务器错误

CodesDescription
500 Internal Server Error客户端请求有效，但服务器处理时发生了意外，未按预期工作，需要稍后再次请求
503 Service Unavailable服务器无法处理请求，一般用于网站维护状态

Content 返回数据格式为 JSON 对象；

提供链接，客户端通过访问 https://api.host.com 可以发现应用 API URL 入口及参数，不查文档，也知道下一步应该做什么；

风格

API JSON 返回数据对象的属性名，单词全小写，单词之间用下划线 _ 分隔，即 snake_case；

URI、URL 路径、路由名、接口名，单词全小写，单词之间用中横线(连词符) - 分隔，即 kebab-case；

  • 日期：_at；created_at，updated_at，closed_at，expires_at
  • 真值性：has_、is_；has_projects，is_verified
  • 链接：_url
  • 唯一性编号：id
  • 编号：_id
  • 状态：state: all|open|closed
  • 类型：type: all|private
  • 计数，有列表项：_count
  • 计数，无列表项：size
  • 列表项：总计数/total_count，数组对象/items，Header 中传递分页信息：每页条数/per_page，当前页码/page
  • 排序：sort: created|updated|[condition field...] & direction: desc|asc

集合上的 GET 请求列表项的数组属性名为 items：

  • GET /customers → items: [] ，客户列表
  • GET /customers/:id/orders → items: [] ，客户订单列表
  • GET /customers/:id/orders/:id/lineitems → items: [] ，客户订单商品列表

应用/APP 和 数据接口/API 之间数据，表示类型、种类，不要用数字代号；

NameTypeDescription
sortstringcreated 或 updated，默认 created
directionstringasc 或 desc，缺少 sort 参数将忽略

  • Status: 200 OK
  • page: 1
  • per_page: 1
