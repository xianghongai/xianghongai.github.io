Commit and Changelog

借助 Git 提交日志以实现版本自动化，并附加 CHANGELOG。

Format of the commit message

type(scope?): subject
body?
footer?

**BREAKING CHANGE: ** 在可选的正文或脚注的起始位置带有 BREAKING CHANGE: 的提交，表示引入了破坏性变更（这和语义化版本中的 MAJOR 相对应）。 破坏性变更可以是任意 类型 提交的一部分。

Header

Type

用于说明 commit 的类别，

  • feat：新功能（feature），(和语义化版本中的 MINOR 相对应)
  • fix：修补 bug，(和语义化版本中的 PATCH 相对应)
  • perf：代码更改以更好的形式实现
  • refactor：重构（即不是新增功能，也不是修改bug的代码变动）
  • style： 格式（不影响代码运行的变动）
  • docs：文档（documentation）
  • test：增加测试
  • ci：CI 配置文件和脚本的更改
  • build：构建过程或辅助工具的变动
  • chore：其它杂项

feat、fix 之类的 Commit 将会出现在 Change Log 中。

Scope

用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同。

Description/Subject

commit 的简短描述，

  • 动词开头，第一人称现在时
  • 第一个字母不要大写
  • 结尾不加句号/点号

Body

commit 的详细描述，可以分成多行；说明代码变动的动机，以及与以前行为的对比。

Footer

用于两种情况；

1. 当前代码与上一个版本不兼容，

Footer 部分以 BREAKING CHANGE 开头，后面是对变动的描述(Before/After)、以及变动理由和迁移方法。

BREAKING CHANGE: isolate scope bindings definition has changed.

    To migrate the code follow the example below:

    Before:

    scope: {
      myAttr: 'attribute',
    }

    After:

    scope: {
      myAttr: '@',
    }

    The removed `inject` wasn't generaly useful for directives so there should be no code using it.

2. 关闭 Issue，

Closes #123, #456, #789

Revert

当前 commit 用于撤销以前的 commit，以 revert: 开头，后面跟着被撤销 Commit 的 Header。

Body部分的格式是固定的，必须写成 This reverts commit <hash>.，其中的hash是被撤销 commit 的完整 SHA 标识符。

revert: chore: add placeholder to firebase example (#7979)
This reverts commit eb6ebf822ff9221e5cdabf24d2dc7d305cae6475.

日志生态系统模块

  • commitlint，检测 Commit message 是否符合格式，

    • 配置 .commitlintrc.json 文件 "extends": ["@commitlint/config-conventional"] 属性
    • 安装 @commitlint/prompt-cli，配置 package.json 属性："scripts": { "commit": "commit" }，CLI 交互式撰写合格的 Commit message
    • 可安装 husky ，添加钩子在提交前检查 Commit 是否符合约定，配置 package.json 属性："husky": { "hooks": { "commit-msg": "commitlint -E HUSKY_GIT_PARAMS" } }，

  • prompt-cli替代方案：commitizen，CLI 交互式撰写合格的 Commit message，

    • 安装 cz-conventional-changelog 适配器，配置 .czrc 文件
    • 配置 package.json 属性："scripts": { "commit": "git-cz"}
    • 项目和构建过程不同，可选择不同的适配器，参考官网说明

  • conventional-changelog，从项目的提交消息和元数据生成 CHANGELOG

    • 可通过 conventional-changelog-cli 生成 CHANGELOG，"scripts": { "changelog": "conventional-changelog -p angular -i CHANGELOG.md -s -r 0 && git add CHANGELOG.md" }
    • 可通过 standard-version 高层次自动化版本控制和生成 CHANGELOG，"scripts": { "changelog": "standard-version" }，npm run changelog -- --help
    • 可通过 standard-changelog，采用 Angular 提交约定方式生成 CHANGELOG，"scripts": { "changelog": "standard-changelog", }，npm run changelog -- --help

  • Visual Studio Code 可安装 Commitizen 扩展以支持交互式提交

  • 示例：https://github.com/karma-runner/karma/blob/master/CHANGELOG.md
  • Angular Commit Message Guidelines: https://github.com/angular/angular/blob/master/CONTRIBUTING.md#-commit-message-guidelines
  • 约定式提交: https://www.conventionalcommits.org/zh/v1.0.0-beta.3/
